# Архитектура проекта Skachki

`ARCHITECTURE.md` — источник правды по структуре проекта.

Перед крупными изменениями нужно сверяться с этим файлом. Если фактическая структура проекта или договорённости устарели, сначала нужно явно сказать об этом и обновить `ARCHITECTURE.md`, затем вносить кодовые изменения.

## Текущее состояние ветки `cleanup`

Проект переведён с большого монолитного `app.js` и временных patch/hotfix-файлов на модульную структуру.

`index.html` больше не подключает `app.js`. Запуск приложения идёт через `js/bootstrap.js`.

Основной интерфейс теперь строится вокруг нижнего меню:

- `Конюшня`
- `Гонки`
- `Разведение`
- `Рейтинг`

Верхние заголовки экранов нужно использовать осторожно, потому что интерфейс Telegram Mini App может перекрывать верхнюю часть приложения.

## Основные правила разработки

1. Не добавлять новые hotfix-файлы поверх основной логики.
2. Не размещать игровую бизнес-логику в `index.html`.
3. Все изменения вносить в соответствующий модуль.
4. Сложные изменения делать небольшими безопасными шагами.
5. Не использовать костыли, если можно исправить архитектурно правильно.
6. Не создавать лишние файлы без необходимости.
7. Если новый файл уменьшает сложность и убирает смешение ответственности — его можно создать.
8. После изменения структуры проекта обновлять этот файл.
9. После каждого изменения проверять GitHub Pages и Telegram Mini App.

## Назначение файлов

### `index.html`

Отвечает за:

- базовую HTML-разметку экранов;
- порядок подключения CSS и JS;
- статические контейнеры экранов;
- базовые модальные окна.

Не должен содержать:

- расчёты характеристик;
- игровую логику;
- генерацию сложных карточек;
- долгие inline-скрипты;
- временные патчи, которые можно вынести в модуль;
- большие inline CSS-блоки.

Текущий статус:

- inline JS экрана рождения жеребёнка вынесен в `js/breeding-render.js`;
- inline CSS Разведения и экрана рождения вынесен в `css/luxury.css`;
- `js/breeding-service.js` подключён перед `js/breeding.js`.

### `css/app.css`

Базовые стили приложения:

- layout;
- экраны;
- базовые кнопки;
- модальные окна;
- общие компоненты.

### `css/luxury.css`

Премиальный визуальный слой:

- карточки лошадей;
- медальоны;
- нижнее меню;
- визуальный стиль Конюшни, Разведения, Гонок и Рейтинга;
- стили экрана рождения жеребёнка;
- стили inline-редактора имени жеребёнка.

Правило:

- визуальные улучшения карточек и экранов делать здесь, если это не базовый layout;
- не дублировать большие CSS-блоки в `index.html`;
- JS-модули не должны создавать `<style>` для постоянных стилей.

### `js/bootstrap.js`

Запуск приложения:

- ожидание DOM;
- Telegram Web App init;
- вызов `SKACHKI_GAME.init()`;
- аварийный вывод ошибок.

### `js/game.js`

Центральный координатор:

- глобальное состояние;
- загрузка/сохранение;
- переходы между экранами;
- общие игровые helper-функции;
- подключение экранных модулей.

Не должен содержать:

- детальную логику разведения;
- детальную логику гонки;
- HTML конкретных карточек.

### `js/data.js`

Справочные данные:

- типы гонок;
- тренировки;
- имена ботов;
- подсказки;
- общие таблицы баланса, если они нужны нескольким модулям.

### `js/horse.js`

Доменная логика лошади:

- создание лошади;
- нормализация старых сохранений;
- расчёт класса;
- форма;
- пол;
- скрытые качества;
- ранги.

Правило:

- всё, что относится к сущности `horse`, должно быть здесь или в общем UI-рендере, а не внутри отдельных экранов.

### `js/state.js`

Сохранение и загрузка состояния:

- `localStorage`;
- миграции сохранений;
- сброс прогресса;
- сериализация.

Правило:

- сервисные родители не должны сохраняться как обычные лошади игрока.

### `js/ui.js`

Общие UI-хелперы и переиспользуемые компоненты:

- поиск DOM-элементов;
- toast;
- переключение экранов;
- общий рендер карточки лошади;
- портреты лошадей;
- звёзды потенциала;
- медальон класса.

Правило:

- если карточка лошади нужна на нескольких экранах, использовать `SKACHKI_HORSE_UI.renderHorseCard`.

### `js/stable.js`

Экран Конюшни:

- список лошадей;
- карточки лошадей;
- подробности лошади;
- действия Конюшни.

### `js/training.js`

Экран Тренировки:

- выбор направления тренировки;
- стоимость;
- применение улучшений;
- связь тренировки с потенциалом, когда эта механика будет доработана.

### `js/breeding-service.js`

Промежуточный модуль сервисных родителей.

Сейчас отвечает за:

- племенного жеребца;
- племенную кобылу;
- цены внешних родителей;
- проверку внешнего родителя;
- список доступных родителей;
- поиск обычного или сервисного родителя.

Текущий статус:

- файл создан и подключён;
- `js/breeding.js` ещё содержит дублирующую сервисную логику;
- следующий безопасный шаг — перевести `js/breeding.js` на использование `SKACHKI_BREEDING_SERVICE` и удалить дубли.

### `js/breeding.js`

Текущий модуль Разведения.

Сейчас содержит слишком много ответственности:

- выбор родителей;
- дублирующая логика племенного жеребца и племенной кобылы;
- стоимость племенной станции;
- расчёт наследования;
- прогноз потомка;
- рендер экрана;
- экран рождения;
- обработчики событий.

Это рабочий файл, но он требует поэтапного рефакторинга.

### `js/breeding-render.js`

Промежуточный модуль визуальных улучшений Разведения.

Сейчас отвечает за:

- улучшение карточки рождённого жеребёнка;
- inline-редактор имени жеребёнка;
- DOM-перестройку карточки результата рождения.

Правило:

- это не финальная структура;
- при дальнейшем рефакторинге сюда нужно переносить рендер Разведения из `js/breeding.js`;
- постоянные стили этого модуля должны оставаться в `css/luxury.css`, а не генерироваться через JS.

Целевая структура после рефакторинга:

```text
js/breeding-service.js
js/breeding-logic.js
js/breeding-render.js
js/breeding.js
```

#### `js/breeding-logic.js`

Будет отвечать за:

- наследование обычных показателей;
- наследование скрытых качеств;
- наследование потенциала;
- расчёт прогноза потомка;
- создание объекта жеребёнка.

#### `js/breeding-render.js`

В финальном виде будет отвечать за:

- рендер выбора родителей;
- рендер блока `Будущий жеребёнок`;
- рендер экрана рождения;
- вспомогательные UI-блоки Разведения.

#### `js/breeding.js`

После рефакторинга должен остаться координатором:

- текущее состояние экрана Разведения;
- выбранные родители;
- переходы между шагами;
- обработчики событий;
- вызовы service / logic / render.

### `js/race-menu.js`

Меню гонок:

- выбор типа гонки;
- выбор лошади;
- проверка готовности старта.

Правило:

- в выборе лошади использовать карточку как в Конюшне;
- пользовательский класс не должен быть главным фильтром/ограничителем.

### `js/race-engine.js`

Сама гонка:

- Phaser;
- трек;
- движение лошадей;
- камера;
- сбои;
- завершение гонки.

### `js/results.js`

Результаты гонки:

- таблица финиша;
- награды;
- обновление статистики.

Правило:

- обычные заезды не должны влиять на статистику лошади;
- статистика должна меняться только там, где это явно предусмотрено правилами режима.

### `js/utils.js`

Общие утилиты без зависимости от игрового состояния.

## Правила карточки лошади

1. Единый источник карточки — `SKACHKI_HORSE_UI.renderHorseCard`.
2. Потенциал отображается звёздами:
   - 50–60: 1 звезда;
   - 60–70: 2 звезды;
   - 70–80: 3 звезды;
   - 80–90: 4 звезды;
   - 90–100: 5 звёзд.
3. Класс отображается цифрой на медальоне.
4. Пол отображается значком.
5. Слова `Класс` на медальоне быть не должно.
6. Карточка должна визуально совпадать в:
   - Конюшне;
   - выборе лошади в Гонках;
   - выборе родителей в Разведении.
7. Если экран рождения использует отличающуюся карточку, она должна следовать тем же правилам: потенциал звёздами, класс цифрой, пол значком.

## Правила Разведения

1. Пользователь выбирает жеребца и кобылу.
2. Если своих доступных жеребцов нет, доступен `Племенной жеребец`.
3. Если своих доступных кобыл нет, доступна `Племенная кобыла`.
4. Сервисные родители:
   - используются только для расчёта потомка;
   - не сохраняются в Конюшню;
   - не получают `offspringCount`;
   - имеют отдельную стоимость.
5. Базовые цены:
   - племенной жеребец — `80` монет;
   - племенная кобыла — `100` монет.
6. Потомок наследует:
   - породу;
   - масть;
   - характер;
   - обычные показатели;
   - скрытые качества;
   - потенциал.
7. Блок `Будущий жеребёнок` должен показывать прогноз, а не точный результат.
8. Прогноз потенциала показывается звёздами.
9. Точные значения открываются только после рождения.
10. У рождённого жеребёнка должна быть возможность задать имя.

## Правила экранов Telegram Mini App

1. Верх интерфейса не должен перекрываться системным UI Telegram Mini App.
2. Основная навигация — нижнее меню.
3. Не нужно дублировать название экрана сверху, если оно уже понятно из нижнего меню или контекста.
4. Важные кнопки не должны прятаться под нижним меню.
5. Скролл должен работать на всех длинных экранах, включая экран рождения жеребёнка.

## Порядок безопасного рефакторинга

1. Вынести временный inline JS из `index.html`. Статус: сделано.
2. Перенести временные стили Разведения из `index.html` и `js/breeding-render.js` в CSS. Статус: сделано.
3. Создать и подключить `breeding-service.js`. Статус: сделано.
4. Перевести `breeding.js` на использование `SKACHKI_BREEDING_SERVICE`. Статус: следующий шаг.
5. Отделить расчёты наследования в `breeding-logic.js`.
6. Перенести HTML-рендер из `breeding.js` в `breeding-render.js`.
7. Последним шагом уменьшить `breeding.js` до координатора.
8. После каждого шага проверять полный цикл Разведения.

## Обязательная проверка после изменений в Разведении

Проверить:

- экран Разведения открывается;
- есть доступный жеребец;
- есть доступная кобыла;
- работает племенной жеребец;
- работает племенная кобыла;
- цена корректно считается;
- кнопка `Скрестить` активируется корректно;
- жеребёнок создаётся;
- имя жеребёнка редактируется;
- жеребёнок появляется в Конюшне;
- сервисные родители не появляются в Конюшне;
- сервисным родителям не увеличивается потомство;
- Конюшня, Гонки и Рейтинг не ломаются.

## Что ещё осталось сделать

- Перевести `js/breeding.js` на использование `SKACHKI_BREEDING_SERVICE`.
- Отделить расчёты наследования в `js/breeding-logic.js`.
- Перенести HTML-рендер из `js/breeding.js` в `js/breeding-render.js`.
- Проверить и удалить старые неиспользуемые файлы, если они ещё лежат в репозитории.
- После стабилизации перейти к балансу и игровому циклу.
